# Plan: Interactive Interface & Stress Testing Improvements

## Executive Summary

This plan outlines enhancements to the web-based visualizer and stress testing framework to improve usability, accuracy, and diagnostic capabilities. The current systems work but need refinement for production-quality toolpath analysis and testing.

---

## 1. Interactive Web Visualizer Improvements

### 1.1 Critical Fixes

**Issue: Arc Rendering Inaccuracy**
- `app.js:142-162` `arcToSegments()` uses fixed 30 steps regardless of arc size
- This can cause visible faceting on large arcs or wasteful rendering on tiny arcs
- **Fix:** Compute steps dynamically based on arc length and tolerance
  - Use chordal error formula: `stepCount = ceil(arcLength / (2 * sqrt(r * tolerance - tolerance^2)))`
  - Or simpler: angular step based on tolerance (Δθ ≈ sqrt(2*tolerance/r))

**Issue: Fragile G-code Parsing in renderOptimized()**
- `app.js:109-131` uses basic regex that fails on complex G-code (F words, Z moves, modal suppression)
- Cannot parse arcs with helical Z moves (when --allow-helix enabled)
- **Fix:** Create a lightweight G-code parser client-side or receive pre-parsed segment data from server
  - Option A: Server returns array of {type: 'arc'|'line', start, end, center, ...}
  - Option B: Include arc rendering parameters in response to avoid client-side parsing

**Issue: Z-coordinate Handling Inconsistent**
- Ideal path uses `p.z + offset` but start point for optimized parsing may have wrong Z
- `renderOptimized()` assumes current.z from startPoint, but startPoint may be degraded's first point (not original)
- **Fix:** Use consistent Z layering: ideal (z=0.05), original (z=0.1), optimized (z=0.15)

**Issue: Error Handling Missing**
- Server API failures are silent; client retains old visualization without user feedback
- **Fix:** Add try-catch around fetch, display toast/notification on errors

### 1.2 Feature Enhancements

**Haas Constraint Controls**
Add UI controls to adjust Haas parameters in real-time:
- Tolerance slider (0.0001 to 0.01 with step 0.0001)
- Min radius input
- Max radius input
- Max IJK input
- Checkbox: Allow Helical Arcs
- Checkbox: Modal Suppression

These would send extra query params to `/api/generate` and `/api/toolpath`.

**Feedrate Visualization**
- Color-code path segments by feedrate (slow=red, fast=blue)
- Show feedrate in tooltip when hovering over path
- Requires server to include feedrate in segment data

**Statistics Panel Expansion**
- Show arc sweep distribution (histogram or min/max/avg)
- Display IJK magnitude warnings
- Show rejected arc count (due to constraints)
- Display chordal deviation heatmap if verification data available

**Comparison Mode**
- Load two different optimization results and overlay them
- Compute differences in arc count, compression ratio
- Useful for tuning parameters

**File Upload Support**
- Allow user to upload custom .nc files instead of only generated test cases
- Server: add POST `/api/upload` endpoint
- Client: file input with drag-and-drop

**Responsive Design**
- UI overlay currently fixed at 300px width
- Make collapsible on small screens
- Adjust canvas container styling

## 2. Stress Testing System Improvements

### 2.1 Framework Refactoring

**Modular Test Cases**
- Current `StressTester.runTest()` takes a generator function but is limited
- Create a formal test case definition structure:
  ```javascript
  {
    name: 'High-Frequency Jitter Circle',
    generator: () => generator.generateCircle(0, 0, 10, 200, { jitter: 0.01 }),
    tolerance: 0.001,
    constraints: { minRadius: 0.1, maxIJK: 10 },
    expectations: { minCompression: 50, maxDeviation: 0.001 }
  }
  ```
- Loop through array of test definitions automatically

**Parallel Test Execution**
- Tests run sequentially; can be slow for large batteries
- Use `Promise.all()` with concurrency limit (e.g., 4 at a time)
- Careful: `ArcFitter` instance not thread-safe; create new instance per test

**Detailed Report Export**
- Currently only console output
- Add `exportReport(format, filePath)` method
- Formats: JSON (full results), CSV (summary), HTML (pretty)
- Include: test parameters, arc statistics, max deviation per test, warnings

**Metrics Collection**
- Track: number of arcs, average sweep, radius distribution, IJK extremes
- Collect data across all tests for aggregate statistics
- Identify patterns: which path types fail constraints most often?

### 2.2 Test Case Expansion

**Add Stress Tests:**
- **Tiny Arcs**: Very small radius near min constraint
- **Huge Arcs**: Radius near max constraint or IJK limit
- **Near-Collinear**: Points almost collinear (ill-conditioned for circle fit)
- **Sweep > 180°**: Full semicircles, major arcs
- **Helical Circles**: Circles with constant Z slope (helix)
- **Mixed Modes**: Interleave G0, G2/G3, M-codes to test modal suppression
- **Incremental (G91)**: Test absolute/relative mode handling
- **Inch vs Metric**: Verify unit conversion (G20 vs G21)

**Add Degradation Profiles:**
- **Low Precision**: Rounding to 0.1mm or 0.01" (simulates old controller)
- **Segment Thinning**: Randomly drop points to simulate coarse sampling
- **Systematic Drift**: Add sinusoidal bias that changes slowly (thermal)
- **Quantization Noise**: Uniform random noise within ±half LSB

**Add Real-World Patterns:**
- **Pocket Perimeter**: Rectangle with filleted corners (4 quarter-circles)
- **Contour Following**: Complex shape with varying curvature
- **High-Frequency Detail**: Small zigzag patterns on otherwise smooth path
- **Zigzag Fill**: Parallel lines with tight turns

### 2.3 Scoring System Enhancement

Current `calculateScore` uses:
- Compression: 40%
- Accuracy: 40% (based on maxDev vs tolerance)
- Stability: 20% (stub)

**Improve Stability Metric:**
- Track arc direction consistency (excessive G2/G3 flipping?)
- Measure radius variance across consecutive arcs (smoothness)
- Count constraint violations weighted by severity

**Add New Dimensions:**
- **Efficiency**: Ratio of arc length to chord length (benefit of arc vs lines)
- **Robustness**: Did optimizer crash or produce invalid arcs?
- **Constraint Compliance**: Percentage of arcs within Haas limits

**Configurable Weights:**
- Allow weights to be passed to `StressTester` constructor
- Different use cases: production (accuracy-heavy), research (compression-heavy)

### 2.4 Regression Test Integration

**Create Regression Suite:**
- Fixed set of test cases that must always pass
- Store expected results (arc count, compression range) in JSON
- Detect performance regressions (runtime increase)
- Detect accuracy regressions (deviation increase)
- Run automatically on CI/CD

**Golden Outputs:**
- For selected seeds, store optimized G-code as "golden" reference
- Use binary diff or line-by-line comparison
- Flag any changes for manual review

## 3. Technical Debt & Bug Fixes

### 3.1 Code Quality Issues

**Inconsistent Arc Validation**
- `ArcFitter.isArcValid` and `Verifier.verify` have nearly identical logic but different implementations
- **Fix**: Consolidate into single `ArcValidator` class/module with static methods
- Use same algorithm to ensure consistency

**Missing JSDoc in Critical Files**
- `StressTester.js` and `TestCaseGenerator.js` lack documentation
- **Fix**: Add comprehensive JSDoc for all public methods

**Root Path Resolution in server.js**
- `path.join(process.cwd(), fileName)` assumes client sends filename relative to cwd
- Security risk: path traversal if user sends `../../../etc/passwd`
- **Fix**: Restrict to `./gcode/` directory or use upload endpoint with sanitized filename

**Random Seed Not Deterministic Across Browsers**
- `TestCaseGenerator` uses custom LCG, but server Date.now() for randomness if no seed
- **Fix**: Ensure all randomness derives from provided seed; document seed contract

### 3.2 API Design Issues

**Inconsistent Response Shapes**
- `/api/toolpath` returns `{original: [...], optimizedStrings: [...]}`
- `/api/generate` returns `{original: [...], groundTruth: [...], optimizedStrings: [...]}`
- **Fix**: Standardize response envelope:
  ```javascript
  {
    success: true,
    data: { original, optimized, groundTruth, summary, arcs? },
    error?: string
  }
  ```

**No Pagination/Streaming for Large Paths**
- Entire toolpath loaded in one HTTP response
- Could OOM for very dense toolpaths
- **Fix**: Implement chunked transfer or WebSocket streaming (future; not urgent)

**CORS Missing**
- Server does not set CORS headers
- Works only on same origin
- **Fix**: Add `app.use(cors())` for development flexibility

### 3.3 Missing Features

**No Progress Indication**
- Stress test with many cases shows no progress
- **Fix**: Add `onProgress` callback to `StressTester.runTest()`
- Update client UI to show progress bar

**No Cancellation**
- Long-running tests cannot be stopped midway
- **Fix**: Add `abort()` method to `StressTester` using AbortController pattern

**No Configuration File**
- Tolerances, constraints must be passed via CLI/flags each time
- **Fix**: Support JSON/YAML config file with defaults

- All new code has ≥80% test coverage

## 6. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Arc rendering still inaccurate after dynamic steps | Medium | Use point-to-arc verification during rendering; allow adaptive tessellation |
| Server load increases with new controls | Low | Parameters are lightweight; no extra computation until optimization |
| Parallel testing causes memory leaks | Medium | Properly dispose of ArcFitter instances; monitor memory |
| Breaking changes to API | High | Version API (v1 → v1.1 is compatible; v2 requires opt-in) |
| Client-side parsing cannot be removed immediately | Medium | Implement as opt-in first, deprecate old param, then remove |
